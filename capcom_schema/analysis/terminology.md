# レポート用語統一ルール（最優先）

> このファイルは `SKILL.md` 配下の最優先ガイドラインです。
> レポート本文・エグゼクティブサマリー・付録・evidence-box・スライドすべてに適用されます。
> 違反は **品質不合格**（執筆のやり直し）となります。

---

## 0. 最重要原則

**レポート受容者（人間の読者）は APOLLO の内部構造を知りません。** 読者にとって `spatial_context`, `cluster_dynamics`, `saturnv_clusters.json`, `data_notes.md`, `quadrant_summary` といった内部識別子は意味不明な英単語に過ぎず、レポートの可読性と信頼性を毀損します。

**この用語表は、内部識別子が外部向けレポートに漏出することを防ぎ、どのツール（Claude Code / Codex / Antigravity）でレポートを生成しても同じ呼称が使われることを保証するために存在します。**

## 4. Mission Objective の扱い — 絶対遵守

`voyager/mission.json` の `mission_objective` フィールドには、ユーザーが VOYAGER / CAPCOM ページで入力した **分析の問い** が格納されている。この原文を**レポートのどこにもベタ貼りしてはいけない**。

### 4-0. 原則

- レポート中の「エグゼクティブサマリー」「本分析の問いと回答」「クロスモジュール統合分析」など、どの章でも Mission Objective の原文をそのまま転記してはいけない
- 執筆者（エージェント）は原文を読解し、「本分析が答えようとしている問い」を**自然な日本語 1-2 行**として書き下すこと
- 原文の語順・箇条書き構造・会話的な語尾（「〜してください」「〜を分析してください」等）・固有名詞の使い方を、そのままコピーしないこと

### NG 例（絶対に避けること）

ユーザー入力 `mission_objective`:
> 「A 社の次の商品開発と、必要であれば M&A 候補の分析を行ってください」

❌ **NG（原文ベタ貼り）**:
> `本レポートは『A 社の次の商品開発と、必要であれば M&A 候補の分析を行ってください』という問いに対し、...`

❌ **NG（語尾だけ削った貼付け）**:
> `本分析のミッション: A 社の次の商品開発と、必要であれば M&A 候補の分析を行う`

✅ **OK（咀嚼して書き下し）**:
> `本レポートは、A 社が次に投入すべき商品領域と、それを実現するための外部連携（M&A を含む提携候補）を特定することを目的としている。`

### 4-A. 各章での適用

| 章 | Mission Objective の扱い |
|---|---|
| エグゼクティブサマリー | 冒頭で「本分析が答えた問い」として咀嚼した 1-2 行を提示し、続けて直接回答（3-5 行）を書く。原文引用は禁止 |
| 本編「本分析の前提」章 | population_meta の設計意図と合わせて咀嚼（設計意図は `query_intent` の項参照） |
| クロスモジュール統合分析 / 結論章 | 「本分析で答えるべき問い」を咀嚼した 1 行で参照。「Mission Objective『...』によれば」のような引用は禁止 |
| 別冊（経営層向け要約版） | 本編と同じく咀嚼した問いを使う。本編と別冊で咀嚼文が食い違うのは品質不合格（本編で使った表現を統一する） |

### 4-B. 咀嚼のガイドライン

1. 原文から**分析対象（企業・技術領域）**・**目的（何を明らかにするか）**・**制約・除外条件**の 3 要素を抽出する
2. 3 要素を「〜を目的としている」「〜の観点で分析する」のような**宣言調の自然な日本語**で再構成する
3. 会話的表現（「〜してください」「〜してほしい」）は排除し、論述調に変換する
4. 余計な修飾語（「必要であれば」等の曖昧な条件）は、そのまま転記せず**分析の観点として書き下す**

## 5. 母集団メタ情報の扱い

`voyager/context.json` の `population_meta` フィールドには、ユーザーが CAPCOM ページで任意入力した以下の情報が含まれます：

| フィールド | 用途（レポート内での扱い） |
|---|---|
| `query_intent`（母集団論理式の設計意図） | **執筆者が咀嚼した上で「本分析の前提」章に自然な日本語として書き下す**（ベタ貼り禁止）。さらに Phase B 以降の**全 deep_dive・クロスモジュール分析・結論章で「分析の視座」として内在化**させ、各章で最低1箇所は意図に照らした解釈を織り込む |
| `query_logic`（母集団論理式） | **付録 D（母集団検索式）に `#raw` ブロックで全文掲載**。検索式は DB 検索のコマンド文字列なので、そのまま原文で掲載して構わない（設計意図のような「咀嚼して書き下す」ルールは適用されない）。本文では「付録に示す検索式で抽出された母集団」と言及する。**さらに Phase A で STOP-GATE A（構造化読解）を経由し、DB 別構文を踏まえた論理構造の把握をユーザーと合意する**（詳細: `analysis/query_logic_reading.md`） |
| `coverage_years`（収録年情報） | **付録 A の「対象期間」欄** に反映する。また時系列分析（ATLAS・ハイプサイクル等）の解釈で「本母集団は {coverage_years} の範囲で収録されており...」と明示する |
| `database_name`（使用した特許データベース名） | **付録 A の「特許データベース」欄** に記載する。また `data_notes.md` のカバレッジ制約注記では、かつての「J-PlatPat 等」に代えて「{database_name} から抽出された特許データ」と表記する。**Phase A では `query_logic_reading.md` §2 の DB 別構文リファレンスを参照して検索式を読解する際の DB 識別に使う** |

### 5-A. `query_intent` の取り扱い — 絶対遵守の 3 原則

**原則 1: ベタ貼り禁止**
- レポートのどこにも `query_intent` の原文をそのまま転記してはならない（本分析の前提章・付録 D・各 deep_dive すべて）
- 「本分析の前提」章では、執筆者が原文を読解した上で、3〜5 行の自然な日本語段落として**書き下す**こと。書き下しの構成要素: ①分析目的、②母集団の輪郭（何を含め何を除外したか）、③どの切り口を重視するか
- 用語・語順・箇条書き構造を原文から流用せず、Mission Objective とも結び付けた「本分析を読むための視座」として提示する

**原則 2: ユーザーとの対話で設計理解を確定させる（STOP-GATE）**
- Phase A で `query_intent` を読み取った段階で、執筆者は自分の言葉で「分析目的・母集団の輪郭・視座」の 3 点を整理し、`AskUserQuestion` ツールで**ユーザーに確認する**
- ユーザーが「この理解で進める / 修正が必要 / 補足を追加」を返すまで次のステップに進めない
- 確定したあとで初めて、その理解を本分析の視座として作業メモに固定する
- AI 側が自動で解釈を確定して進めるのは禁止（設計意図の解釈は分析の根幹に関わるため、必ずユーザー合意を経由する）
- `query_intent` が未指定の場合はこの対話ゲートを省略してよい（その旨をユーザーに 1 行で報告）

**原則 3: 全分析を通じた視座として機能させる**
- Phase B の Evidence 精読・クロスモジュール分析・Phase C の各 deep_dive・Phase D の結論章すべてで、確定した視座を分析の視座として内在化させる
- **指定章で明示的な意図参照を義務化**（下記 5-A-1）: エグゼクティブサマリー冒頭 / 各 deep_dive 冒頭 / クロスモジュール分析の結論段 / 戦略的提言 / 仮説検証サマリー
- 意図に沿った論点を優先的に掘り下げる
- 意図と整合しない結果（除外したはずの領域が部分的に混入している等）は、本文に分散させずに **Phase D で独立した章「分析過程で確認された追加的事項」** にまとめる（`analysis/report_structure.md` §7 参照）
- 設計意図を無視した汎用的な分析は品質不合格

### 5-A-1. 指定章での意図参照の義務化（強度「中」）

以下 5 つの章では、**「本分析の視座」「設計意図に照らすと」「当初の分析観点では」等の参照表現を最低 1 箇所**含めること。検出は Phase D gate `Check 12` で自動実施（本文内の意図参照語をカウント）:

| 章 | 義務化される意図参照の例 |
|---|---|
| **エグゼクティブサマリー冒頭** | 「本分析の視座（〜）に即して答えると、〜が明らかとなった」（宣言調、問い形式禁止） |
| **各 deep_dive 冒頭** | 「本章は {視座} の観点から、〜を読み解くものである」 |
| **クロスモジュール分析の結論段** | 「本分析の視座に照らすと、この結果は〜を示唆している」 |
| **戦略的提言** | 「本分析の視座から導出される提言として、〜」 |
| **仮説検証サマリー** | 「各仮説は、本分析の視座との整合性の観点で検証された」 |

### 5-A-2. サブクエスチョン化（Phase A で執筆者の作業メモ用として作成）

Phase A で `query_intent` を 3 点整理（分析目的・母集団の輪郭・視座）した後、**その 3 点整理をさらに具体的な分析観点として 3-5 個に分解**し、`reports/_phase_a_decisions.json` の `sub_questions` フィールドに保存する。

**重要制約（絶対遵守）**:
- サブクエスチョンは **執筆者（エージェント）の作業メモ専用**
- **レポート本文には「問い / 答え」「Q1 / A1」等の形式で書いてはいけない**
- レポート本文は通常の**宣言調の論述**で書く
- 「Q1: 最大の成長領域はどこか？」のような問い文、「A1: X 技術ポケットである」のような回答文は品質不合格

**サブクエスチョンの役割**:
1. 執筆者が Phase B/C/D で「この分析は何を明らかにすべきか」を見失わないための内部指針
2. Phase D の結論章で、意図由来の論点が漏れなくカバーされたかの自己点検
3. Phase D gate `Check 12` で、サブクエスチョンの **キーワード** が本文に登場するかを自動検証

**Phase A での作成手順**:
```
Step 1: query_intent の 3 点整理（分析目的・母集団の輪郭・視座）をユーザーと合意
   ↓
Step 2: 執筆者が 3 点整理を分解し、「本分析が明らかにすべき具体的観点」を
         3-5 個の箇条書きとして起草（問い形式で起草してよい。レポート本文ではない）
   ↓
Step 3: 各サブクエスチョンから主要キーワード 1-3 個を抽出
   ↓
Step 4: AskUserQuestion でサブクエスチョン一覧 + キーワード抽出結果を提示
   ↓
Step 5: ユーザーが「この分解で進める / 修正 / 追加」を選択
   ↓
Step 6: 確定結果を _phase_a_decisions.json の sub_questions に保存
```

**例**:

ユーザー入力 `query_intent`:
> 「CNF 構造材料で自社がどの技術ポケットに注力すべきかを決める」

執筆者が Phase A で起草するサブクエスチョン（作業メモ用）:

```json
{
  "sub_questions": [
    {"id": "SQ1", "content": "本母集団で最も成長している技術領域はどこか", "keywords": ["成長", "技術領域"]},
    {"id": "SQ2", "content": "本母集団で自社がまだ手をつけていない空白領域はどこか", "keywords": ["空白領域", "自社"]},
    {"id": "SQ3", "content": "主要競合（旭化成・東レ）が注力している領域はどこか", "keywords": ["競合", "旭化成", "東レ", "注力"]},
    {"id": "SQ4", "content": "参入障壁が低い / 高い領域の構造", "keywords": ["参入障壁", "低い", "高い"]},
    {"id": "SQ5", "content": "3-5 年後のリーダー候補領域", "keywords": ["リーダー候補", "中期"]}
  ]
}
```

**レポート本文での扱い（❌ NG / ✅ OK）**:

❌ **NG（問い / 答え形式）**:
> **Q1: 最大成長領域はどこか?**
> A1: 本母集団ではタイヤ用ゴム複合材料領域が CAGR +254% で最大成長している。

❌ **NG（「問い」への直接言及）**:
> 本分析の問い「自社が注力すべき領域はどこか」に対しては、〜

✅ **OK（宣言調の通常論述、意図参照あり）**:
> 本分析の視座である「自社の注力領域選定」に即して成長動向を精査すると、タイヤ用ゴム複合材料領域が CAGR +254% と本母集団内で群を抜いて伸長している。一方で自社の出願はこの領域に集中しておらず、{視座} の観点から見て重要な空白領域として位置づけられる。

✅ **OK（エグゼクティブサマリーでの宣言調回答）**:
> 本分析の視座に即して答えると、自社が優先的に注力すべき技術領域は以下 3 つである。第一に、〜

**いずれも空文字（未指定）の場合**は、以下のフォールバック表記を使用：
- `query_intent` 未指定 → 「本分析の前提」章の「分析の視座」サブセクションごと省略。全 deep_dive での「視座に照らした解釈」も省略してよい
- `query_logic` 未指定 → 付録 D ごと省略（「非公開」等と書かない）
- `coverage_years` 未指定 → `context.json` の `dataset.period`（出願年の最小〜最大から自動算出）を使用
- `database_name` 未指定 → 「提供された特許データセット」と一般化表記

### 5-B. `query_logic`（母集団検索式）の構造化読解 — Phase A STOP-GATE A

`query_logic` が指定されている場合、執筆者は検索式を単に付録 D にコピペするだけでなく、**Phase A の STOP-GATE A で構造化して読解し、ユーザーと合意する**。

**Phase A で実施する 4 ステップ**（詳細は `analysis/query_logic_reading.md` §1）:
1. **DB 識別**: `database_name` があれば使用、未指定なら構文特徴（`/TX` なら J-PlatPat、`HTX=` なら JP-NET、`$Wn` なら PatSnap 等）から推測
2. **構文分解**: AND/OR/NOT で節に分け、「分類条件 / キーワード条件 / 出願人条件 / 日付条件 / その他」に仕分け
3. **意図推定**: 各条件について「なぜこの条件がここにあるか」を推定
4. **ユーザー確認**: `AskUserQuestion` で DB 名・分解結果・意図推定を提示、「この読解で合っているか」を確認

**続く STOP-GATE B（意図 ↔ 論理 整合性検査）**: `query_intent` と `query_logic` の両方が指定されている場合、8 項目チェックリスト（詳細は `query_logic_reading.md` §4）で乖離を検出し、Critical/Warning/Info の 3 段階で分類。改善提案を添えてユーザー確認。

**続く STOP-GATE C（データ側からの逆読み）**: 全ケースで実施。patents.csv から算出した実態（出願人/IPC/年分布/HHI 等）と、設計意図・検索式の整合性を確認。

これら 3 つの STOP-GATE は Phase A の必須手続きであり、省略不能（`SKILL.md ## 0. 絶対遵守ゲートルール` 第 1-2 項）。詳細手順は `analysis/query_logic_reading.md` § 6 の統合フローを参照。

## 6. スコープ限定ルール — 「本母集団」と「業界全体」を混同させない（絶対遵守）

> **関連**: 本節は全タイプ共通の基本防壁。**母集団タイプ（A/A'/B/C/D）ごとに追加の表現禁止ルール**が `analysis/population_type_metrics.md` §3 にある。両者を併せて適用すること。
> - 本 §6: 全タイプで「本母集団では〜」等の限定修飾を必須化
> - `population_type_metrics.md` §3: タイプ B/C/D で「市場集中」「業界シェア」等の特定表現を追加禁止

### 6-0. なぜこのルールがあるのか

APOLLO の分析対象は常に **ユーザーが検索式で絞り込んだ特許母集団** であり、**業界全体・市場全体・技術分野全体** ではない。レポートを読む人間の読者は検索式を暗記していないので、主語を曖昧にしたまま書くと、以下のような読み違えが発生する：

- 本母集団の中では A 社が 1 位 → 読者は「業界で A 社が 1 位」と誤解する
- 本母集団では出願件数が増加 → 読者は「市場全体が成長している」と誤解する
- 本母集団では技術 X が主流 → 読者は「業界の標準技術は X」と誤解する

この誤読は経営判断を致命的に誤らせる（特に別冊）。本節のルールは、レポート全体を通じて「どこまでが母集団内の観察か」「どこからが母集団外への一般化か」を読者が瞬時に判別できるようにするための構造的防壁である。

### 6-1. 禁止表現（修飾なしの主語無限化）

以下の表現を「本母集団では／本分析では／本対象特許群では」等の限定修飾なしに使うのは **品質不合格**：

| 禁止表現 | 理由 |
|---|---|
| 「業界では〜」「業界全体では〜」 | 業界 ≠ 本母集団 |
| 「市場では〜」「市場全体で〜」 | 市場 ≠ 本母集団（市場は特許ではなく売上・シェア） |
| 「産業では〜」「当該産業では〜」 | 産業 ≠ 本母集団 |
| 「業界の標準／主流／トレンド」 | 本母集団の観察は業界標準を直接示さない |
| 「全体として〜」（単独使用） | 何の全体か不明。「本母集団全体として」等に限定する |
| 「技術分野では〜」「この技術領域では〜」 | 技術分野 ≠ 本母集団（別の検索式なら別の結果） |
| 「企業の間では〜」「各社は〜」（無限化） | 本母集団に含まれる出願人に限定する必要 |
| 「主要プレイヤーは〜」（単独） | 「本母集団の主要出願人は〜」等に限定 |
| 「日本では〜」「グローバルでは〜」（地理的一般化） | データベースが地域限定なら地理的一般化は不可 |

### 6-2. 必須表現（母集団限定を明示する修飾）

上記の無限化表現の代わりに、以下のいずれかを必ず文頭・文中に含めること：

| 限定修飾（推奨） | 使用場面 |
|---|---|
| 「本母集団では〜」 | 一般的な限定。万能 |
| 「本分析の特許群では〜」「本対象特許群では〜」 | 「特許」の限定を強調したい場合 |
| 「本検索式で抽出された範囲では〜」 | 検索式の限界を強調したい場合 |
| 「本データセット内では〜」 | データセットの限界を強調したい場合 |
| 「{database_name}（{coverage_years}）の範囲では〜」 | 具体的データベース名・期間を明示 |
| 「本分析の対象（{total_patents} 件）では〜」 | 件数スコープを明示 |
| 「本母集団に含まれる出願人では〜」「本母集団の上位 N 社では〜」 | 出願人ランキングを論じる場合 |

### 6-3. 例外: 業界全体への一般化が許容される 2 条件

以下のいずれかを満たす場合に限り、**母集団外への一般化表現**（「業界では」「市場では」等）を使ってよい：

**条件 A: Web 調査・外部データでの裏付けあり**
- Web 調査で得た市場規模・業界統計・他社決算・政府統計などを `#footnote[...]` で引用した上で、その情報源に基づく一般化として書く
- 例: 「世界 CNF 市場は CAGR 17.5% で成長している#footnote[Grand View Research (URL), 取得日: 2026-04-14]」は許容（これは本母集団の観察ではなく外部データの引用）

**条件 B: 母集団が業界全体を十分カバーしていることが明示的に論証されている**
- `population_meta.query_intent` や `database_name` の設計から、母集団と業界全体のカバレッジがほぼ一致することが明示的に示されている場合（例: 業界団体が定義する全出願を網羅している等）
- この場合も初出時に「本母集団は業界の{概算カバレッジ}%をカバーしており、以降『業界』と『本母集団』を同義として扱う」等の定義を置くこと
- 定義なしに暗黙で同義化するのは禁止

**上記 2 条件に該当しない場合は、必ず「本母集団では〜」に限定表現すること。**

### 6-4. NG / OK 例

#### 例 1: ランキング

❌ **NG（無限化）**:
> 業界ではトヨタ自動車が 1 位である。

❌ **NG（無限化）**:
> 主要プレイヤーはトヨタ・ホンダ・日産の 3 社である。

✅ **OK（母集団限定）**:
> 本母集団の主要出願人は、トヨタ自動車（152 件）・ホンダ技研工業（98 件）・日産自動車（76 件）の順で、上位 3 社で全体の 34.2% を占める。

#### 例 2: 成長率

❌ **NG（無限化）**:
> 市場では 2020 年以降 CAGR +15% で成長している。

✅ **OK（母集団限定）**:
> 本母集団では、2020 年以降の出願件数が CAGR +15.2%/年で伸びている（2020 年 234 件 → 2023 年 358 件）。

✅ **OK（外部裏付けあり）**:
> 本母集団では 2020 年以降の出願件数が CAGR +15.2%/年で伸びており、これは外部市場調査（CAGR +17.5%#footnote[Grand View Research (URL), 取得日: 2026-04-14]）と整合的であり、研究開発投資が市場成長に呼応していることを示唆する。

#### 例 3: 技術トレンド

❌ **NG（無限化）**:
> 業界では SiC パワー半導体が主流技術として確立しつつある。

✅ **OK（母集団限定 + 解釈）**:
> 本母集団では、SiC 関連特許（クラスタ 3）が 2021 年以降に急増しており（CAGR +47%）、主要出願人（ロームほか 3 社）が同領域に集中投資している様子が観察される。

#### 例 4: 地理的一般化（データベースが日本限定）

❌ **NG（データベース制約を無視した一般化）**:
> 日本では CNF 技術の研究開発が活発化している。

✅ **OK（データベース制約を明示）**:
> 本母集団（{database_name} から抽出された日本出願）では、CNF 関連出願が 2020 年以降に加速しており、日本国内の研究開発が活発化している可能性を示唆する。ただし米国・欧州・中国の出願動向は本母集団に含まれておらず、グローバルな比較は Web 調査等で別途補う必要がある。

#### 例 5: 出願人の戦略

❌ **NG（無限化）**:
> A 社は次世代半導体領域で先行している。

✅ **OK（母集団限定）**:
> 本母集団において、A 社は次世代半導体関連クラスタ（クラスタ 0・3・7）で合計 128 件（母集団全体の 8.2%）を占め、母集団内の他社より先行している。

### 6-5. 各章での適用ルール

| 章 | スコープ明示の粒度 |
|---|---|
| 本分析の前提 | 「本分析の範囲と限界」サブセクションで母集団の輪郭を明示（→ `report_structure.md` §3） |
| エグゼクティブサマリー | 冒頭で「本分析は〜{database_name}〜の範囲を対象としている」を 1 行明示。以降の主張は母集団限定表現で統一 |
| 各モジュール deep_dive | セクション冒頭（または最初の evidence-box 内）で「本母集団において」等の限定を 1 回は明示。それ以降は自然な文脈で引き継いでよい |
| クロスモジュール分析 | 各仮説→検証→結論の「結論」段で母集団限定を明示 |
| 戦略的提言 | 提言が母集団内観察に基づくか、外部情報を加味した一般化かを明示的に区別 |
| 経営層向け要約版（別冊） | エグゼクティブサマリーに加え、各「発見」「推奨アクション」でもスコープ明示を徹底（→ `executive_summary_guide.md` §3-6） |
| 付録 | 付録 A の対象範囲欄で母集団の輪郭を再掲。`query_logic` が指定されていれば付録 D に全文掲載 |

### 6-6. 自動チェック（Phase D `phase_d_gate.sh` Check 10）

Phase D の品質ゲートスクリプトは、以下の条件で自動検出する：

- 無限化語（「業界では」「市場では」「産業では」「業界全体」「市場全体」「全体として」「業界の」「市場の」「主流」「標準技術」等）の出現数
- 限定語（「本母集団」「本分析の」「本対象特許群」「本検索式」「本データセット」等）の出現数
- **合格条件**: 無限化語の出現数 ≤ 限定語の出現数 × 0.3 （= 限定語がはるかに多いこと）
- これを超える場合、レポート全体の主語スコープが甘い可能性が高く品質不合格

### 6-7. まとめ（執筆前のチェックポイント）

1. 段落を書く前に「この段落で主張していることは、**本母集団の観察** なのか **業界全体への一般化** なのか」を明確にする
2. **本母集団の観察** なら、段落の冒頭または主文に「本母集団では〜」等の限定修飾を必ず置く
3. **業界全体への一般化** を書きたいなら、Web 調査で外部データを `#footnote` で引用するか、母集団のカバレッジ論証を行った上で書く
4. どちらでもないなら書かない（推測の一般化は品質不合格）
5. **母集団タイプ（A/A'/B/C/D）を確認**: Phase A STOP-GATE C で確定したタイプを `reports/_phase_a_decisions.json` で確認し、タイプ B/C/D の場合は `population_type_metrics.md` §3 の追加禁止表現リスト（「市場は寡占」「業界シェア」「市場構造」等）も適用する

## 8. まとめ（執筆前に必ず確認）

1. レポート本文は「人間の読者」向けであり、APOLLO 内部の JSON 構造を一切見せてはいけない
2. 内部ファイル名・フィールド名・ガイドファイル名は全て本表の正式呼称に置換する
3. モジュール名（Saturn V, MEGA, ATLAS, Explorer, CREW, EAGLE, NEBULA, CORE）とブランド機能名（TELESCOPE, PROBE, PULSE）は、初出時の補足を伴って使用可能

## 2. deep_dive 全文コピールール

deep_dive.typの内容をreport.typにコピーする際の**最重要ルール**:

- **要約・圧縮・省略は一切禁止**
- 全セクション・全段落・全テーブルをそのまま転記する
- 許可される変更:
  1. 見出しレベル調整（`==`→`===`等）
  2. 冒頭の`#import`削除（report.typで1回だけimport）
  3. ヘッダ/フッタコメント削除のみ
- deep_dive.typが200行なら、report.typの対応章もほぼ200行であるべき

## 3. 各章の執筆ルール

### 本分析の前提（⚠️ 章・任意情報のみで構成）

`voyager/context.json` の `population_meta` 配下の任意入力情報をもとに、分析の前提条件を簡潔に提示する。**入力されたフィールドのみを記載し、未入力フィールドはセクションごと省略する。**

#### ⚠️ 設計意図（`query_intent`）の取り扱い — 絶対遵守

**設計意図はベタ貼り（コピペ）禁止。** 執筆者は以下3つの責務を負う:

1. **Phase A でユーザーと対話確認する**: `query_intent` を読み取ったら、執筆者は「分析目的・母集団の輪郭・視座」の 3 点を自分の言葉で整理し、`AskUserQuestion` ツールでユーザーに提示・確認する。ユーザーが「この理解で進める / 修正が必要 / 補足を追加」で回答を返すまで次に進まない（詳細は `SKILL.md` Phase A の `query_intent` STOP-GATE 参照）
2. **分析全体で汲み取る**: 確定した視座を Phase B 以降の全 deep_dive・クロスモジュール分析・結論章で「分析の視座」として内在化させる。意図に沿った論点を優先的に掘り下げ、意図と整合しない分析結果が出た場合も **「意図の観点では〜」「意図と照らすと〜」** の形で必ず言及する。設計意図を無視した汎用的な分析は品質不合格
3. **本章では咀嚼して書き下す**: 「本分析の前提」章では `query_intent` を**そのまま転記せず**、Phase A でユーザーと合意した視座を元に **「本分析が何を明らかにしようとしているか」「なぜこの母集団を選んだか」「何を除外したか」を 3〜5 行の自然な日本語段落**として書き下す。ユーザー入力文の文体・構造をそのままコピーするのは禁止

記載テンプレート:
```typst
= 本分析の前提

本分析は、以下の母集団・条件のもとで実施された。

#styled-table(
  columns: (1fr, 3fr),
  header: ("項目", "内容"),
  // population_meta.database_name が指定されている場合のみ以下の行を追加
  [使用特許データベース], [{database_name}],
  // population_meta.coverage_years が指定されている場合のみ
  [収録年情報], [{coverage_years}],
  // それ以外の場合は context.json の dataset.period を用いる
  [出願年範囲], [{period}（出願年基準）],
  [対象件数], [{total_patents}件],
)

// population_meta.query_intent が指定されている場合のみ、以下のブロックを追加
// ⚠️ 必ず執筆者が咀嚼した自然な日本語で書く。query_intent の文字列をそのまま貼らない
=== 分析の視座

{query_intent を読解し、以下の3要素を自然な日本語 3-5 行で書き下す:
  1. 本分析が何を明らかにしようとしているか（分析目的）
  2. 母集団の輪郭（何を含め、何を除外したか）
  3. 結果をどう解釈する観点か（どの切り口を重視するか）
 設計意図の原文のコピー・言い換えだけで済ませず、Mission Objective とも結び付けた「本分析を読むための視座」として提示する}

// population_meta.query_logic が指定されている場合は、付録 D で全文掲載することを明示
#text(size: 10pt, style: "italic")[
  母集団の検索式は付録 D に全文掲載している。
]

// ⚠️ 以下のサブセクションは全レポートで必須（任意項目ではない）
=== 本分析の範囲と限界

本レポートに記載される出願件数・成長率・出願人シェア等の数値は、いずれも
上記の母集団内での観察値であり、業界全体・市場全体の傾向を直接示すものでは
ない。以下の点に特に留意されたい。

- **本母集団 ≠ 業界全体**: 本母集団は {database_name or "提供された特許データセット"} から
  {query_logic が指定されていれば「付録 D に示す検索式」、未指定なら「条件検索」} で
  抽出された {total_patents} 件であり、当該技術分野の全出願を網羅するものではない。
  異なる検索式・異なるデータベースを使った場合、異なる結果が得られる可能性がある
- **地理的カバレッジの限界**: 本母集団は {database_name or "提供された特許データセット"} の
  収録範囲に限定される。収録範囲外の地域（例: 米国・欧州・中国）の出願動向は
  本母集団には含まれず、グローバルな比較には Web 調査等の外部情報を要する
- **「本母集団では」という前置きの意味**: 以降の本文で「本母集団では」「本分析の特許群では」
  等の限定表現が現れた場合、それは「本母集団内の観察」を意味し、業界全体の
  傾向を断定するものではない
- **業界全体への一般化が必要な場合**: 本レポートでは Web 調査で外部データを引用した
  箇所に限り、業界・市場全体への一般化を行っている。その場合は本文中に脚注
  （#footnote[...]）で出典を明記している
// nebula_strategy.selected_mode == "omit" の場合のみ、以下を追加
// （"execute" / "web_compensation" の場合は不要）
- **本分析は特許情報のみを対象としている**: 学術論文・ニュース・政策動向といった
  外部環境データは本分析の対象外である。業界動向・市場規模・政策動向の確認は
  別途の調査で補完する必要がある
```

### モード別の本章テンプレート差分

`nebula_strategy.selected_mode` に応じて「本分析の範囲と限界」サブセクションの記述を調整する:

| モード | 追加注記 |
|---|---|
| `execute` | 追加注記なし（NEBULA 章で外部環境は既にカバー） |
| `web_compensation` | 「外部環境については Web 調査により補完している。詳細は『外部環境分析』章および脚注を参照」旨を 1-2 行で追加 |
| `omit` | 上記テンプレート末尾の「本分析は特許情報のみを対象としている」注記を 2-3 行で明記（Phase D gate Check 13 で自動検証） |

**設計意図の書き下しの具体例**:

> ユーザー入力 `query_intent`:
>「本母集団はCNF（セルロースナノファイバー）関連技術のうち、食品・化粧品用途を除外し、構造材料・複合材料用途に焦点を絞って抽出した。IPC C08L, C08K を主軸に、B29, B32 のコーティング関連を補助的に含めている。」

❌ **NG（ベタ貼り）**:
> *本母集団は CNF（セルロースナノファイバー）関連技術のうち、食品・化粧品用途を除外し、構造材料・複合材料用途に焦点を絞って抽出した。IPC C08L, C08K を主軸に、B29, B32 のコーティング関連を補助的に含めている。*

✅ **OK（咀嚼した書き下し）**:
> *本分析は、CNF 関連技術のうち**構造材料・複合材料としての応用可能性**を焦点として位置づけている。そのため、食品添加物や化粧品用途といった別系統の応用は母集団から意図的に除外されており、結果として得られる技術動向はマトリクス・コンポジット領域に偏る設計となっている。IPC 主軸は高分子複合系（C08L/C08K）に置きつつ、成形・表面処理（B29/B32）を補助的に含めることで、材料設計から加工までのバリューチェーンを視野に入れている点が本分析の特徴である。以降の各モジュール分析は、この「構造材料としての CNF」という視座で解釈されるべきである。*

**重要**:
- いずれの任意情報も `population_meta` で空文字の場合は**その行またはセクションを省略**する
- 執筆者が勝手にデータベース名（J-PlatPat 等）を補わない
- **設計意図は全分析を通じた「視座」**として機能すること（単発の前提記載で完結させない）
- 各 deep_dive 章・クロスモジュール分析・結論章で、設計意図に照らした解釈を最低 1 箇所は織り込むこと
- 設計意図と整合しない結果（例: 除外したはずの領域が部分的に混入している等）が見つかった場合は、その事実を隠さず指摘し、意図との乖離として記述すること

### NEBULA（環境分析）章 — nebula_strategy モード別の執筆ガイド

`reports/_phase_a_decisions.json` の `nebula_strategy.selected_mode` に応じて、NEBULA 関連章の扱いが 3 パターンに分岐する:

#### モード `execute`（NEBULA 実行済み）

- 現状通り `nebula_deep_dive.typ` を全文コピー
- 学術-特許クロス分析は `cross_module.md` P13 に従って実施
- 仮説検証サマリーでは NEBULA 由来仮説（H1-H5 等）を通常通り検証

#### モード `web_compensation`（Web 調査で補完）

**章タイトル**: `= 外部環境分析（Web 調査に基づく）`（NEBULA という内部モジュール名は使わない）

**必須カバー 4 カテゴリ**:
1. **市場規模・業界統計**（例: Grand View Research、MarketsandMarkets の市場予測）
2. **政策・規制動向**（例: 経産省・厚労省・EPA 等の公表情報）
3. **学術動向・キーパーソン**（例: Google Scholar・主要研究機関の発表）
4. **主要企業動向**（例: 各社 IR・プレスリリース・M&A ニュース）

**執筆要件**:
- 各カテゴリにつき最低 1 つの Web 調査結果を `#footnote[出所, URL, 取得日]` 付きで記述
- `data_notes.md` §3 の Web 情報出所記載ルール（`#footnote[...]` 統一）を遵守
- 付録 C「Web 調査出所一覧」で各カテゴリの出所を集約
- 学術-特許クロス分析の代わりに「外部環境と特許動向の整合性」という形で本文中で言及（cross_module.md P13 の代替）

**記載例**:
```typst
= 外部環境分析（Web 調査に基づく）

本母集団の出願件数成長（CAGR +18%）は、この市場成長予測と整合的である。
```

#### モード `omit`（省略）

- 本章を丸ごと削除
- 「本分析の前提」章の「本分析の範囲と限界」サブセクションで特許情報のみの対象であることを明記（§3 のテンプレートに従う）
- 学術-特許クロス分析も省略
- Phase C の Step 0（NEBULA deep_dive）も省略
- 仮説検証サマリーでは NEBULA 由来仮説が存在しないため、特許データ由来の仮説のみ検証

### エグゼクティブサマリー

- 全 deep_dive の統合要約
- **Mission Objective への回答を冒頭に配置する。ただし原文をベタ貼りしない**（詳細は `analysis/terminology.md` §4 参照）:
  - ❌ NG: `*Mission Objective:* 原文そのまま` のような引用ボックス
  - ❌ NG: 「『...を分析してください』という問いに対し、...」（ユーザー入力の語尾・構造をそのままコピー）
  - ✅ OK: 「本レポートは、〜を特定することを目的としている。結論として、〜」（咀嚼した宣言調で書き下し）
- `exec-summary[...]` で囲む
- 「本分析の前提」章で示した設計意図が指定されている場合、エグゼクティブサマリー冒頭でその意図に照らした結論を 2-3 行で記載する
- **⚠️ スコープ明示の義務化**: エグゼクティブサマリー内の全主張（ランキング・成長率・集中度・技術トレンド等）は「本母集団では〜」「本分析の特許群では〜」等の限定修飾を必ず含めること。業界全体への一般化を書く場合は Web 調査の外部裏付け（`#footnote[...]`）を付すこと（詳細は `analysis/terminology.md` §6、`analysis/common_framework.md` §1「各 Layer のスコープ明示ルール」参照）
- **⚠️ 設計意図への明示的参照（必須）**: エグゼクティブサマリーの冒頭段落で **「本分析の視座（〜）に即して答えると、〜」** のように設計意図に直接呼応する宣言調の文を必ず含める。`reports/_phase_a_decisions.json` の `query_intent_summary` と `sub_questions` の内容をカバーするよう執筆する
  - ❌ NG（問い/答え形式）: 「Q1: 最大成長領域は? A1: 〜である」
  - ❌ NG（原文転記）: 「『自社が注力すべき領域はどこか』という本分析の問いに対して〜」
  - ✅ OK: 「本分析の視座である『自社の注力領域選定』に即して答えると、優先すべき 3 領域は〜」
  - 詳細: `analysis/terminology.md` §5-A-1（指定章での意図参照義務化）、§5-A-2（問い/答え形式の禁止）

### KPIダッシュボード
- `kpi-dashboard(cols: 3, kpi-card(...), kpi-card(...), ...)` で**ページまたぎ防止**
- `kpi-card("ラベル", "値", note: "補足")` で各KPIを表示
- **`cols` パラメータは 3 を厳守。4 以上は禁止**（A4幅で1カードあたり37mm以下になり、長い日本語の値テキストが文字単位で改行されて崩れる）。KPI が9個以上ある場合は `kpi-dashboard(cols: 3, ...)` を複数ブロックに分割する
- **ドル記号の禁止**: kpi-card の value / テーブルセル内で `$` や `\$` を使用禁止。通貨は「ドル」「USD」で表記する
- value テキストの長さに応じてフォントサイズが自動調整される(report_style.typ の `kpi-card` 関数が context で動的計算)。3文字以下=24pt、5文字以下=20pt、7文字以下=16pt、10文字以下=14pt、それ以上=12pt

### クロスモジュール統合分析（最低80行必須）
- 複数のdeep_diveを横断する洞察を新規執筆する（→ `analysis/cross_module.md`）
- NEBULA環境分析と特許分析の統合を含む
- **各パターンは「仮説→検証→結論」の3段構造で15-20行ずつ記述すること。1-2行の箇条書きで済ませるのは品質不合格**
- **⚠️ 各パターンの「結論」段で設計意図への参照を必須化**: 「本分析の視座に照らすと、この結果は〜を示唆している」のような宣言文を含める。`sub_questions` のうちその結論が貢献する観点を執筆者メモで意識しつつ、レポート本文には問い/答え形式ではなく通常の論述として書く

### 分析過程で確認された追加的事項（v8 新設・条件付き必須）

**位置づけ**: 仮説検証サマリーの直後、戦略的提言の直前に配置。

**必須条件**: 以下のいずれかに該当する場合、本章を **必ず** 設ける。該当がない場合は省略可（ただし執筆者が「該当なし」と自己判断した経緯を `_phase_a_decisions.json` の `user_notes` にメモ）:

1. Phase A STOP-GATE B で検出された **意図 ↔ 論理の乖離**（Critical / Warning / Info のいずれか）があり、ユーザーが「このまま進めて範囲と限界に明記」を選択した
2. Phase A STOP-GATE C で **想定外のデータ偏り** が検出され、ユーザーが「このまま進めて明記」を選択した
3. Phase B/C/D の分析過程で、設計意図からは予期されなかったが重要度の高い観察があった（例: 設計意図では除外したはずの領域が部分的に混入していた／意図と直接関係ない強い傾向が観察された）

**執筆ルール**:

- **章タイトルは「分析過程で確認された追加的事項」で固定**。修辞的なタイトル（「驚きの発見」「想定外の衝撃」等）は禁止
- 本章は 20-50 行程度。長すぎず簡潔に
- 各事項を `#note-box[...]` または `#styled-table` で整理
- 各事項について以下 3 点を記述:
  1. **観察内容**: 何が見出されたか（数値根拠付き）
  2. **設計意図との関係**: 当初の分析視座とどう異なるか / 補完するか
  3. **本分析での扱い**: 提言章に反映したか、今後の課題として位置づけたか
- 「驚きの」「衝撃の」等の修辞表現は使わず、冷静で中立的な記述に徹する

**記載例（タイプ B: 競合限定、意図は EV 戦略比較）**:

```typst
= 分析過程で確認された追加的事項

#note-box[
  本分析の視座（3 社の EV 戦略比較）とは別に、分析過程で以下の事項が確認された。
  これらは当初の分析観点には含まれていなかったが、意思決定の参考となるため記録する。
]

== 水素関連出願の混入

本母集団では EV 関連特許を対象としたが、{社名} の出願 32 件（全体の 4.8%）は水素燃料電池関連であり、
IPC H01M8 の指定により結果的に取り込まれた。これは当初の視座（EV = 電池駆動）の想定外であるが、
同社が電動化を幅広く捉えている戦略的示唆として解釈できる。本分析では EV 比較の軸には含めず、
戦略的提言の補足事項として言及する。

== X 社の出願急増の背景

X 社の 2024 年出願件数が前年比 +180% と急増しているが、これは当初の分析視座で想定した
技術領域ではなく、M&A 後の権利統合によるものであることが Web 調査で判明した。
したがって技術動向としての解釈には慎重を要する。
```

**章の省略基準**:
- 上記 3 条件のいずれにも該当しない場合は省略可
- ただし省略する場合、`_phase_a_decisions.json` の `user_notes` に「追加的事項なし（Phase A/B/C/D で特記すべき想定外観察なし）」と明記する

**品質チェック**:
- 章タイトルが「分析過程で確認された追加的事項」と完全一致しているか（修辞なし）
- 各事項に数値根拠があるか
- 「観察内容 / 設計意図との関係 / 本分析での扱い」の 3 点が記述されているか

## 4. 結論章（= 戦略的提言）の執筆ガイド

- report.typの最終本文章として必須。4サブセクション構成
- `#conclusion-box("タイトル")[本文]` で主要結論をハイライト
- `#recommendation-card("高", "タイトル", "説明", timeframe: "短期(6ヶ月以内)")` で優先度付き推奨
- `#action-items("アクション1", "アクション2", ...)` で具体的ToDoリスト
- 推奨アクションは最低5件、各アクションに優先度（高/中/低）と時間軸（短期/中期/長期）を明記
- 全ての提言にはdeep_diveの分析結果からの論理的根拠を記載すること
- **⚠️ 設計意図への直結性（必須）**: 戦略的提言のサブセクション「分析結果の総括」冒頭で「本分析の視座から導出される結論として、〜」のような宣言文を含める。`sub_questions` のすべての観点が結論章を通してカバーされるよう執筆する（執筆者メモで対応確認）
- **⚠️ サブクエスチョン・キーワードのカバレッジ**: `_phase_a_decisions.json` の各 `sub_questions.keywords` は結論章の本文（総括 + 戦略的インプリケーション + 推奨アクション）に **すべて登場** すること。Phase D gate `Check 12` で自動検証される
- **⚠️ 問い/答え形式の禁止**: 「Q1 / A1」「問い 1 への回答」等の記法は禁止。通常の宣言調の論述で書く（詳細: `analysis/terminology.md` §5-A-2）

## 5b. 付録D: 母集団検索式テンプレート（条件付き）

`voyager/context.json` の `population_meta.query_logic` が空文字でない場合のみ、付録 D を追加する:

```typst
== D. 母集団検索式

本分析で用いた母集団は、以下の検索式によって抽出されている。

#raw(
  lang: "text",
  "{query_logic をそのまま #raw ブロックに埋め込む。改行・特殊文字を保持}"
)
```

**重要**:
- `query_logic` は `#raw` ブロック（コードブロック）として Typst に埋め込み、整形されない生テキストのまま掲載する（検索式は **DB 検索のコマンド文字列** なので、そのまま原文で掲載して構わない）
- 検索式中に Typst の特殊文字（`$`, `[`, `]`, `#`, `*`, `<`, `>`, `@` 等）が含まれていても、`#raw(...)` が自動でエスケープするため手動処理不要
- `query_logic` が **指定されていれば**、付録 D 節を必ず追加し、本文のエグゼクティブサマリーでも「母集団の詳細は付録 D に示す」と言及する
- `query_logic` が **未指定** の場合は、付録 D 節ごと省略する（「非公開」等と書かない）
- 検索式が長大（数十行以上）の場合でも全文掲載する。省略は不可
- ⚠️ **設計意図（`query_intent`）は付録 D には転記しない**。設計意図は「本分析の前提」章で執筆者が咀嚼した自然な日本語として書き下すものであり、原文のベタ貼りはレポートのどこにも登場させないこと

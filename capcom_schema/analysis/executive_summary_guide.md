# 経営層向け要約版（別冊）執筆ガイド

> このファイル名はレポート本文に書かないこと（執筆者の内部参照専用）。
> 生成トリガー: Phase A で `AskUserQuestion` により「別冊も生成する」とユーザーが選択した場合のみ
> 出力先: `reports/report_executive.typ` → `typst compile` で `reports/report_executive.pdf`

---

## 0. 目的と位置づけ

### 本編と別冊の関係

| 項目 | 本編（`report.typ`） | 別冊（`report_executive.typ`） |
|------|-------------------|---------------------------|
| 想定読者 | 分析担当者・事業部長・技術企画 | 経営層（役員会・意思決定者） |
| 想定ページ数 | 60〜120 ページ | **8〜12 ページ（厳守）** |
| 目的 | 根拠と詳細の完全提示 | **意思決定に必要な核心のみ** |
| 手法詳細（SBERT/UMAP 等） | 詳細掲載 | **省略（用語解説にも含めない）** |
| ミクロ分析（個別特許引用） | 15 件以上 | **3〜5 件の厳選のみ** |
| クロスモジュール分析 | 3 パターン以上各 15-20 行 | **最重要 2-3 パターンを 3-5 行で要約** |
| 読了時間 | 60〜90 分 | **15 分以内** |

### 🔑 別冊の本質

**別冊は「本編の刈り取り版」ではない。**「経営判断に必要な核心を、本編とは独立に再構成した凝縮版」である。以下の違いが本質:

- ❌ NG（単なる刈り取り）: 本編の各章から段落をコピーし、半分に削ったもの
- ✅ OK（エッセンス凝縮）: 全分析から「意思決定に資する洞察」を抽出し、**本分析の Mission Objective と `query_intent` から導かれる「今回の意思決定テーマ」に即して再構成**したもの

## 1. 生成トリガー（Phase A STOP-GATE）

### タイミング

ユーザーが「レポートを書いて」「report.typ を生成して」等と依頼した際、Phase A の最初（用語統一・population_meta 確認の直後、データ精読の前）に `AskUserQuestion` で以下を確認する:

```
質問: 本編レポート（60-120ページ）に加えて、経営層向け要約版（別冊、8-12ページ）も生成しますか？

選択肢:
- ✅ 両方生成する（本編 + 別冊）— 推奨: 経営層への提出を想定している場合
- 📘 本編のみ — 詳細分析のみで十分な場合
- ❓ 相談したい — 別冊の位置づけや粒度を議論してから決定
```

**ユーザーが応答するまで次のフェーズに進まない。** AI 自己判断で「不要と思います」「省略します」と決めるのは禁止。

### 別冊生成が選択された場合

- 作業メモに「別冊生成フラグ = ON」として記録
- Phase D で本編生成後、続けて別冊を生成する
- **両方の品質ゲート**（`phase_d_gate.sh` および別冊用チェック）を通過する必要がある

### 別冊不要が選択された場合

- 通常通り本編のみ生成
- ただしレポート完成後、「経営層向け要約版を後から追加することも可能です」と 1 行案内する

## 3. 執筆ルール — エッセンス凝縮の方法

### 3-1. 「So What」原則

別冊の各段落は **「経営判断に何を意味するか（So What）」** を必ず含む。「〜が検出された」「〜が分析された」という記述的表現は禁止。

❌ NG: 「Saturn V TELESCOPE 分析の結果、43 個のクラスタが検出された。」
✅ OK: 「技術領域は 43 個のサブドメインに分岐しており、単一戦略で全域をカバーするのは非現実的。本分析ではこのうち 3 つの成長リーダー・クラスタ（全体の 28% をカバー）への集中投資を推奨する。」

### 3-2. 視座の再構成（キー）

**別冊を構造化する「意思決定テーマ」は、毎回 Mission Objective と `query_intent` から導き出す**。定型の分類軸を機械的に当てはめるのは不可。

**判断の流れ**:
1. Mission Objective と `query_intent` を読み、本分析が**具体的に何の意思決定に資するか**を 1 行で書き出す
   - 例: 「CNF 構造材料領域で、自社がどの技術ポケットに注力すべきかを決める」
   - 例: 「次世代パワー半導体で、ライバル X 社にどう対抗するか戦略を固める」
   - 例: 「本テーマから撤退すべきか、継続投資すべきかを判断する」
2. その意思決定テーマに直結する**発見・推奨アクションのみ**を別冊に残す（本編の全項目を網羅する必要はない）
3. 章立ても意思決定テーマに沿って再構成する。§2 のテンプレートは一般形であり、実際の章タイトル・順序はテーマに応じて調整してよい

**参考: よくある意思決定テーマ**（必ずしもこれに限らない。本分析に即して選択する）

- 投資領域の優先順位付け
- 撤退・縮小判断
- 提携・M&A の候補探索
- 競合に対する防衛・攻撃戦略
- 研究開発テーマの選定
- 技術ロードマップの見直し

**重要**: 上記は「思考の起点」であり、**全テーマを網羅する必要はなく、無関係なテーマを無理に盛り込むのは禁止**。例えば「技術動向の俯瞰」が主目的なら、M&A や人材配分に触れる必要はない。各発見・推奨アクションは、本分析で実際に導出された結論のみに基づくこと。

### 3-3. 凝縮の具体技法

| 本編の要素 | 別冊での扱い |
|----------|------------|
| クラスタ構造の詳細（43 個全て解説） | **上位 3-5 個のみ**、しかも本分析の意思決定テーマに直結する視点で再構成 |
| 手法パラメータ（min_cluster_size=15 等） | **一切言及しない** |
| 個別特許ミクロ分析（15 件以上） | **3-5 件の代表特許のみ**、かつ意思決定テーマを象徴する特許に限定 |
| 仮説検証サマリー | **検証結果のみ**を 1 テーブルで（仮説導出過程は省略） |
| 用語解説（B 付録） | **別冊には含めない**（必要な用語は本文中で短く補足） |
| クロスモジュール分析（3 パターン × 15-20 行） | **最重要 2 パターンを各 3-5 行で要約** |

### 3-4. 数値の取捨選択

- **残す**: 総件数、主要クラスタ件数、主要出願人件数、CAGR（成長率）、主要シェア、市場規模
- **削る**: HHI / Entropy / Gini などの統計指標詳細、ノイズ率、次元削減パラメータ、前処理統計

本編にある数値のうち、**「経営会議で即答できる数字」のみ**を別冊に残す。

### 3-5. ビジュアルの絞り込み

本編で使った snapshot を別冊に転用する場合、**最大 3 枚まで**。選定基準:
1. クラスタ動態マップ（4 象限）または MEGA PULSE 4 象限（戦略ポジションが一目でわかる）
2. ハイプサイクル（技術ライフサイクル位置が一目でわかる）
3. 共起ネットワークまたは AI ランドスケープ（技術の広がりが一目でわかる）

各ビジュアルには **キャプション + 1 行の "Takeaway"** を必ず付記する。

## 4. 本編と別冊の関係 — 相互参照ルール

- 別冊の各発見・推奨アクションには、可能であれば **「本編 §{章名} 参照」** の形で本編の該当章を示す
- 本編には別冊への参照は不要（本編は自己完結的に読めるため）
- 両者とも **同じ Mission Objective と同じ結論**を持つ。結論が食い違うのは品質不合格
- 別冊は本編と **同じ `voyager/context.json` の `population_meta`** を使用する（データベース名・収録年・設計意図の再解釈は行わない）

## 5. 禁止事項（品質不合格）

1. **単なる刈り取り**: 本編の段落をコピーして短縮しただけのもの。必ず「経営層視点で再構成」する
2. **手法削除だけ**: 「SBERT / UMAP 等の記述を削除しただけ」の別冊は不合格。凝縮＝削除ではない
3. **ページ数超過**: 13 ページ以上は不合格。削減すること
4. **ページ数不足**: 7 ページ以下は核心が不足しているとみなし不合格
5. **結論の不整合**: 本編と別冊で結論が食い違う、もしくは別冊にしかない結論が登場する
6. **数値の不整合**: 本編と同じ数値を使うこと。別冊独自の再集計は不可
7. **タイトルが本編と完全一致**: 別冊は必ずサブタイトルに「— 経営層向け要約版」を追加
8. **手法詳細の混入**: 「SBERT ベクトルの次元削減により…」のような記述は別冊に不要
9. **内部識別子の漏出**: 本編と同じ `terminology.md` のルールを遵守（`spatial_context`, `cluster_dynamics` 等の露出禁止）
10. **⚠️ スコープ無限化（最重要・経営判断誤導リスク）**: 「業界では〜」「市場では〜」「主要プレイヤーは〜」等の無限化表現を、外部裏付けなしに使うこと。本母集団の観察を業界全体の傾向として読ませる表現は、経営層に誤った前提で意思決定させるため、本編以上に厳格に排除する。全ての主張には「本母集団では〜」「本分析の対象（{total_patents} 件）では〜」等の限定修飾を必ず付すこと（詳細: `terminology.md` §6）

## 5-A. スコープ明示の徹底（経営層別冊・最優先チェック）

経営層は本編の付録や検索式を読まない。別冊の数字を「業界の実態」と受け取り、翌週の経営会議で議題化する。従って別冊におけるスコープ誤読は **本編のそれよりも致命的** である。以下を徹底せよ：

### 5-A-1. エグゼクティブサマリー冒頭のスコープ注記

- §2 のテンプレート通り、`#note-box[...]` で「本分析の対象範囲」を 3-4 行明示する
- データベース名・対象件数・「業界全体の傾向を直接示すものではない」旨を含む
- この注記を省略するのは品質不合格

### 5-A-2. 各「発見」での区別

各「発見」セクションでは、以下の 2 種類を明確に区別して書く：

| タイプ | 書き方 |
|---|---|
| **A. 本母集団内の観察** | 「本母集団では〜」「本分析の対象（{total_patents} 件）では〜」を主文に含める。外部裏付けなしで書ける |
| **B. 外部裏付け付きの一般化** | 「外部市場調査によれば〜#footnote[...]」「業界統計では〜#footnote[...]」のように脚注で出典を明示 |

**両者を混同する書き方は禁止**。

❌ **NG（スコープ無限化）**:
> 発見 1: 業界では EV 向けパワー半導体が主流技術となっている。旭化成はこの領域のリーダーである。

✅ **OK（母集団限定 + 外部裏付けの分離）**:
> 発見 1: 本母集団において、EV 向けパワー半導体関連クラスタが出願件数の 32% を占め、旭化成が母集団内のシェア 18% で最大の出願人である。外部市場調査によれば同領域は世界全体で年 15% 成長しており#footnote[Grand View Research (URL), 取得日: 2026-04-14]、本母集団の出願集中は市場成長と整合的である。

### 5-A-3. 推奨アクションでのスコープ区別

推奨アクションは、その根拠が「本母集団内観察」なのか「外部情報を加味した一般化」なのかを明示的に書く：

❌ **NG**:
> 推奨アクション 1: 業界トップ 3 との提携を検討する

✅ **OK**:
> 推奨アクション 1: 本母集団の上位出願人（A 社・B 社・C 社）のうち、B 社とは技術領域の補完性が高く、提携候補として優先度が高い。ただし本母集団は {database_name} の範囲に限定されるため、グローバルプレイヤーの動向は別途 Web 調査で補った上で最終判断すること。

### 5-A-4. 経営層別冊の追加チェック項目

別冊完成時、§6 の通常チェックに加えて以下を確認：

```bash
# スコープ注記（本分析の対象範囲）の存在確認
grep -c "本分析の対象範囲\|本母集団\|本分析の対象\|本分析の特許群" reports/report_executive.typ
# 合格基準: 5 件以上（エグゼクティブサマリー冒頭 + 各発見 + 各推奨アクション）

# 無限化語の過剰使用チェック
grep -cE "業界では|市場では|産業では|業界全体|市場全体|全体として|業界の主流|業界の標準" reports/report_executive.typ
# 参考: 外部裏付け（#footnote）なしで多用していないか確認
```

## 6. Phase D での生成手順

### 手順

1. 本編 `reports/report.typ` を先に完成させる
2. `bash capcom_schema/scripts/phase_d_gate.sh` で本編が合格していることを確認
3. 別冊生成フラグが ON の場合、以下を実行:
   - 本編の **エグゼクティブサマリー + 結論 + 推奨アクション + 主要 KPI + 最重要 snapshot** を再読する
   - 本ガイド §2 のテンプレートに沿って `reports/report_executive.typ` を生成
   - 本ガイド §3 の執筆ルールに従い、単なる刈り取りではなく再構成する
4. 別冊の品質を以下のコマンドで確認:

   ```bash
   # 行数チェック（別冊は 250-500 行が目安。本編の 10-20% 程度）
   wc -l reports/report_executive.typ

   # 内部識別子の混入チェック（本編と同じ用語ルール）
   grep -nE "(saturnv_clusters|mega_momentum|nebula_[a-z]+|atlas_statistics|explorer_[a-z]+|eagle_[a-z]+|core_classification|crew_network)\.json" reports/report_executive.typ
   grep -nE "\b(spatial_context|cluster_dynamics|noise_analysis|quadrant_summary|time_series|hype_cycle_phase|macro_events|npl_gap_analysis)\b" reports/report_executive.typ
   grep -nE "(SKILL|common_framework|data_notes|cross_module|deep_dive_guide|report_structure|quality_checklist|terminology|map_reading|patent_citation|executive_summary_guide)\.md" reports/report_executive.typ

   # サブタイトルに「経営層向け要約版」が含まれているか
   grep "経営層向け要約版" reports/report_executive.typ

   # 手法詳細の過剰混入チェック（SBERT/UMAP/HDBSCAN の多用は凝縮不足のサイン）
   grep -c "SBERT\|UMAP\|HDBSCAN\|min_cluster_size\|n_neighbors" reports/report_executive.typ
   # 合格基準: 3 件以下（用語解説的な参照のみ）
   ```

5. ユーザーに完成を報告:
   ```
   本編: reports/report.typ（{N1} 行）
   別冊: reports/report_executive.typ（{N2} 行、経営層向け要約版）

   PDF 化:
     typst compile reports/report.typ reports/report.pdf
     typst compile reports/report_executive.typ reports/report_executive.pdf
   ```

### 同時生成の効率化

別冊は本編の後に生成するが、本編生成中に「どの発見・推奨アクション・ビジュアルが最重要か」を **メモしておく** ことで、別冊生成時の迷いを減らせる。Phase C の deep_dive 執筆時に「これは別冊にも入れる」と意識すると良い。

## 7. 執筆時の心構え

経営層向け要約版を書くときは、以下を自問すること:

- 「役員が 15 分でこの冊子を読み、翌週の経営会議で議題として出す」場面を想定できているか?
- 「So What」が各段落で明示されているか?
- 専門用語を知らない経営層が理解できるか?（用語解説なしで）
- 具体的な数値・企業名・技術名が含まれているか?（抽象論で終わっていないか）
- 本編を読まなくても「何を決めるべきか」が分かるか?

この 5 点をクリアしていれば、別冊として合格品質である。

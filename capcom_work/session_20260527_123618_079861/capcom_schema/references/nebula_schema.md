# NEBULA スキーマ

## 対象ファイル
- `data/nebula_hype_cycle.json`
- `data/nebula_macro_events.json`

## モジュール概要
NEBULA (Network Exploration of Business, Users, Law, and Academia) は非特許文献（論文・ニュース・政策）を統合した環境分析モジュール。特許データとNPL（Non-Patent Literature）のギャップを分析する。
- **Hype Cycle**: 特許出願（Patent）、学術論文（Academic）、ビジネスニュース（News）の年次件数推移を比較し、技術のライフサイクル上の位置づけやタイムラグを可視化する。
- **Macro Events**: 政策文書（Policy）と市場レポート（Market）を時系列で整理し、規制動向や市場環境の変化を把握する。

## JSONスキーマ

### Hype Cycle (`nebula_hype_cycle.json`)

#### metadata

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `module` | string | 固定値 `"NEBULA"` |
| `type` | string | 固定値 `"hype_cycle"` |

#### patent_trend

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `"{year}"` | int | 該当年の特許件数。キーは年を表す文字列（例: `"2020"`）、値は整数の件数 |

年ごとの特許出願件数を格納する辞書。分析期間（Global Filter適用後）の全年をカバーし、データがない年は `0` で埋められる。

#### academic_trend

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `"{year}"` | int | 該当年の学術論文件数。キーは年を表す文字列、値は整数の件数 |

NPLデータのうち `data_sub_type == "Academic"` に該当するレコードの年次件数。NPLデータが未アップロードの場合は空の辞書 `{}` となる。

#### news_trend

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `"{year}"` | int | 該当年のビジネスニュース件数。キーは年を表す文字列、値は整数の件数 |

NPLデータのうち `data_sub_type == "Business"` に該当するレコードの年次件数。NPLデータが未アップロードの場合は空の辞書 `{}` となる。

#### 構造例

```json
{
  "metadata": {"module": "NEBULA", "type": "hype_cycle"},
  "patent_trend": {"2018": 45, "2019": 62, "2020": 78, "2021": 91, "2022": 103},
  "academic_trend": {"2018": 12, "2019": 18, "2020": 25, "2021": 30, "2022": 28},
  "news_trend": {"2018": 5, "2019": 8, "2020": 15, "2021": 22, "2022": 19}
}
```

### Macro Events (`nebula_macro_events.json`)

#### metadata

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `module` | string | 固定値 `"NEBULA"` |
| `type` | string | 固定値 `"macro_events"` |
| `count` | int | `items` 配列の要素数（最大30件） |

#### items（配列）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `date` | string | イベントの日付。`parsed_date` が存在する場合は `"YYYY-MM-DD"` 形式、存在しない場合は `unified_date` の文字列または `"-"` |
| `year` | int \| string | イベントの年。整数または `"N/A"`（年情報が欠損している場合） |
| `type` | string | イベント種別。`"Policy"` または `"Market"` |
| `title` | string | イベントのタイトル。`unified_title` カラムの値 |
| `source` | string | 出典・情報源。`unified_source` カラムの値。不明な場合は `"-"` |
| `content` | string | イベントの概要・内容。`unified_content` カラムの値を最大300文字で截断 |

#### 構造例

```json
{
  "metadata": {"module": "NEBULA", "type": "macro_events", "count": 3},
  "items": [
    {
      "date": "2022-06-15",
      "year": 2022,
      "type": "Policy",
      "title": "改正特許法施行",
      "source": "特許庁",
      "content": "改正特許法が施行され、AI関連発明の審査基準が明確化された..."
    },
    {
      "date": "2021-03-01",
      "year": 2021,
      "type": "Market",
      "title": "半導体市場レポート 2021",
      "source": "調査会社A",
      "content": "世界半導体市場は前年比25%成長し..."
    },
    {
      "date": "-",
      "year": 2020,
      "type": "Policy",
      "title": "知的財産推進計画2020",
      "source": "内閣府",
      "content": "知的財産戦略本部が策定した推進計画..."
    }
  ]
}
```

## 解釈ガイドライン

### ハイプサイクルの読み方
- **Patent（Reality）**: 技術の実装・製品化段階を反映する。特許出願は研究成果が事業化に結びつくフェーズの指標。
- **Academic（Research）**: 基礎研究・応用研究段階を反映する。論文発表は技術の黎明期〜成長期の先行指標。
- **News（Buzz）**: 社会・市場の関心度を反映する。ニュース件数は期待のピーク（ハイプ）や幻滅期を可視化する。
- **タイムラグの分析**: 一般的に Academic -> Patent -> News の順で件数が増加する傾向がある。この時間差が「研究から社会実装までのリードタイム」を示す。タイムラグが短い技術は社会実装が加速している可能性がある。
- **トレンドの乖離**: 3つのトレンドが同方向に動いている場合は技術が成熟期に向かっていることを示唆する。Academic は増加しているが Patent が停滞している場合は、研究段階に留まっている可能性がある。News が先行して急増し Academic/Patent が伴わない場合は、ハイプ（過剰期待）の兆候と読める。
- **空の辞書 `{}`**: NPLデータが未アップロードの場合、`academic_trend` や `news_trend` は空辞書となる。この場合、特許データ単体でのトレンド分析となり、クロス分析は不可能。

### マクロイベントの読み方
- **Policy イベント**: 法律・規制・政策の変更を示す。特許出願戦略に直接影響を与える可能性がある（例: 審査基準変更、新規制導入）。
- **Market イベント**: 市場動向・業界レポートを示す。技術の商業化可能性や市場規模の変化を把握するために使用する。
- **時系列の照合**: マクロイベントの時期と patent_trend の変動を照合することで、「特定の政策変更が特許出願行動に影響を与えたか」を分析できる。
- **content の截断**: 300文字で截断されているため、詳細な内容確認が必要な場合は元データ（`df_npl`）を参照すること。
- **最大30件の制限**: トークンオーバーフロー回避のため、上位30件に制限されている。フィルタ期間内のイベントが30件を超える場合は、新しいイベントから優先して格納される（年降順ソート済み）。

## レポートでの典型的な言及パターン

### 環境分析（ハイプサイクル）
- 「特許出願は **{year_range}** の期間で年平均 **X件** であり、{増加/減少/横ばい}傾向にある」
- 「学術論文は **{peak_year}** 年にピーク（**X件**）を迎え、その **N年後** に特許出願のピークが確認された。研究から実装までのリードタイムは約 **N年** と推定される」
- 「ビジネスニュースの急増は **{year}** 年に始まり、特許出願に先行していることから、市場の期待が技術開発を牽引している構図が読み取れる」

### NPLと特許のクロス分析
- 「Academic と Patent のトレンドが同時に上昇しており、研究と事業化が並行して進む成熟技術であることが示唆される」
- 「Academic は堅調に増加しているが Patent は停滞しており、基礎研究は活発であるものの事業化への転換が課題となっている可能性がある」
- 「News のピークが Academic/Patent に先行しており、典型的なハイプサイクルのパターンを示している。今後、幻滅期を経て生産性の安定期に移行するかが注目点である」

### マクロ環境のインパクト分析
- 「**{year}** 年の政策変更（{title}）以降、特許出願件数が **X%** 増加しており、規制環境の変化が出願行動に影響を与えた可能性がある」
- 「市場レポート（{title}）は **{keyword}** 分野の成長を予測しており、同分野の特許出願トレンドとも整合する」
- 「分析期間内に **Policy: X件、Market: Y件** のマクロイベントが確認された。特に **{year}** 年前後にイベントが集中しており、技術トレンドの転換点と一致する」

### データ不足時の注記
- 「NPLデータが未アップロードのため、ハイプサイクル分析は特許データのみに基づく。Academic/Newsトレンドとのクロス分析を行うには、Mission Control で非特許文献データを追加する必要がある」
- 「Policy/Market データが見つからなかったため、マクロ環境分析はスキップされた」

## 新規ファイル: 学術ランドスケープ

### 対象ファイル
`data/nebula_academic_clusters.json`

### 概要
NEBULA環境分析モジュールのAcademic（論文）データに対して、Saturn Vと同様のSBERT+UMAP+HDBSCANクラスタリングを実施した結果。

### 構造
```json
{
  "type": "nebula_academic_landscape",
  "total_papers": 500,
  "n_clusters": 12,
  "noise_count": 35,
  "clusters": [
    {"cluster_id": 0, "label": "[0] keyword1, keyword2, keyword3", "count": 45}
  ],
  "cluster_dynamics": {
    "x_axis": "cumulative_count",
    "y_axis": "cagr",
    "cagr_window_years": 3,
    "clusters": [...],
    "quadrant_summary": {...}
  }
}
```

### フィールド説明
| フィールド | 説明 |
|-----------|------|
| `total_papers` | 分析対象の学術論文総数 |
| `n_clusters` | 検出された研究テーマクラスタ数 |
| `noise_count` | どのクラスタにも属さない論文数 |
| `clusters[].label` | c-TF-IDFによる自動ラベル（NPLストップワード適用） |
| `cluster_dynamics` | クラスタ動態マップ（Saturn Vと同一構造） |

### 解釈上の注意
- ラベリングにはNPLストップワード（英語学術用語を含む）が適用されている
- CAGR計算期間は特許（5年）より短い3年がデフォルト（論文のサイクルが短いため）
- Saturn V特許クラスタとの対比（P13パターン）が最重要の分析観点

### クロスモジュール分析パターン P13
学術クラスタと特許クラスタを対比する際のチェックリスト:
1. 学術で大きいが特許で小さいクラスタ → 研究先行、特許化の機会
2. 特許で大きいが学術で小さいクラスタ → 実用化進展、学術基盤が薄い可能性
3. 両方で成長リーダーのクラスタ → 最も有望な技術領域
4. 学術で新興だが特許でニッチ → 将来のブレイクスルー候補

### ⚠️ データの非対称性に関する重要注意

特許データとNPLデータは**カバレッジが根本的に異なる**。ギャップ分析の際に必ず考慮すること:

| データ | 言語 | カバレッジ | 典型的な偏り |
|--------|------|----------|-----------|
| 特許 | **提供データセット依存** | ユーザーが用意した特許データベースの仕様に依存 | 提供元が国内限定なら米欧中の出願を含まないなど、グローバルの一部しか捕捉できない |
| 学術論文 | **英語が主体** | グローバル（OpenAlex 等） | 世界中の研究を捕捉。特許より母集団が大きい |
| ニュース | **言語依存** | 収集元に依存 | 日本語ニュースのみだと海外報道がカバーされない |

**結果として**:
- 「論文は多いが特許が少ない」は、データカバレッジの違いである可能性が高い。「特許化が遅れている」と即断せず、「提供された特許データセットの範囲では特許化が確認されていない」と記述する
- 件数の直接比較（特許500件 vs 論文2000件）は避ける。**トレンドの方向性**（成長/停滞/衰退）で比較する
- ハイプサイクルでの比較は**ピーク時期のずれ（タイムラグ）**に注目する
- レポートには「本分析の特許データは提供された特許データセット（`voyager/context.json` の `population_meta.database_name` に入力があればその名前）に限定されている」旨の注記を必ず付す。特定のデータベース名（J-PlatPat 等）を執筆者が勝手に補うことは禁止

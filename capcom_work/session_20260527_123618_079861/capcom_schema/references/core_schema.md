# CORE スキーマ

## 対象ファイル
`data/core_classification.json`

## モジュール概要
CORE (Contextual Operator & Rule Engine) は、論理式ベースのルール分類モジュールである。ユーザーが定義した分類軸（技術分類・課題分類・解決手段分類など）ごとに、AND/OR/NEAR/ADJ演算子を用いた論理式ルールで特許を分類する。このJSONには、分類ルールの定義と各カテゴリのヒット件数が格納される。

## JSONスキーマ

| フィールドパス | 型 | 説明 |
|---|---|---|
| `metadata` | object | メタデータオブジェクト |
| `metadata.module` | string | 固定値 `"CORE"` |
| `metadata.total_patents` | integer | 分類対象となった特許の総件数 |
| `metadata.axes` | array of string | 分類軸名のリスト（例: `["技術分類", "課題分類", "解決手段分類"]`） |
| `rules` | object | 分類ルールの本体。キーは分類軸名 |
| `rules.<軸名>` | object | 各分類軸のカテゴリ集合。キーはカテゴリ名 |
| `rules.<軸名>.<カテゴリ名>` | object | 個別カテゴリの情報 |
| `rules.<軸名>.<カテゴリ名>.rule` | string | CORE論理式。AND(`*`)、OR(`+`)、NEAR/ADJ演算子を使用。例: `"(CO2 + 二酸化炭素) * (膜 + メンブレン)"` |
| `rules.<軸名>.<カテゴリ名>.definition` | string | カテゴリの定義・説明文。空文字列の場合あり |
| `rules.<軸名>.<カテゴリ名>.count` | integer | このカテゴリに分類された特許の件数 |

### 「その他」カテゴリについて
いずれのルールにもマッチしなかった特許は「その他」として自動追加される（件数が1件以上の場合のみ）。

| フィールドパス | 値 |
|---|---|
| `rules.<軸名>.その他.rule` | `"(未分類)"` （固定文字列） |
| `rules.<軸名>.その他.definition` | `"いずれのルールにもマッチしなかった特許"` （固定文字列） |
| `rules.<軸名>.その他.count` | 未分類特許の件数 |

### 具体例

```json
{
  "metadata": {
    "module": "CORE",
    "total_patents": 500,
    "axes": ["技術分類", "課題分類"]
  },
  "rules": {
    "技術分類": {
      "CO2分離膜": {
        "rule": "(CO2 + 二酸化炭素) * (膜 + メンブレン)",
        "definition": "CO2分離のための膜技術に関する特許",
        "count": 85
      },
      "触媒変換": {
        "rule": "(触媒 + catalyst) * (変換 + 転換 + コンバージョン)",
        "definition": "触媒を用いたCO2変換技術",
        "count": 62
      },
      "その他": {
        "rule": "(未分類)",
        "definition": "いずれのルールにもマッチしなかった特許",
        "count": 45
      }
    },
    "課題分類": {
      "コスト削減": {
        "rule": "(コスト + 費用 + 経済性) * (削減 + 低減 + 低コスト)",
        "definition": "コスト削減に関する課題",
        "count": 70
      }
    }
  }
}
```

## 解釈ガイドライン

### count値の注意点
- `count` は `df[軸名] == カテゴリ名` の完全一致でカウントされる。1件の特許が複数カテゴリにマッチした場合、その特許の分類値はセミコロン区切り（例: `"CO2分離膜;触媒変換"`）となり、個別カテゴリのcountには含まれない。
- そのため、全カテゴリの `count` の合計は `metadata.total_patents` と一致しない場合がある（複合分類の特許が存在する場合）。
- 「その他」の件数が多い場合は、ルールのカバレッジが不足していることを示す。フェーズ3の再分析機能で追加ルールを検討すべきである。

### 軸の構成
- 分類軸の数と名前はユーザーが自由に定義できる。典型的には「技術分類」「課題分類」「解決手段分類」の3軸だが、これに限定されない。
- `metadata.axes` の配列でどの軸が存在するかを確認できる。

### 論理式の読み方
- `+` は OR（いずれかを含む）
- `*` は AND（すべてを含む）
- `nearN` はN文字以内の近傍（順序不問）
- `adjN` はN文字以内の隣接（順序指定）
- `()` で優先順位を制御
- 論理式は前処理済みテキスト（NFKC正規化・小文字化・図表参照除去済み）に対して評価される

## レポートでの典型的な言及パターン

### 技術ポートフォリオの構成比分析
各分類軸のカテゴリ別件数から、対象母集団の技術構成を定量的に記述できる。例: 「技術分類では『CO2分離膜』が85件(17.0%)で最多であり、次いで『触媒変換』62件(12.4%)が続く」。

### 「その他」比率によるカバレッジ評価
「その他」カテゴリの比率が分類ルールの網羅性を示す指標となる。例: 「その他比率が9.0%(45件)であり、主要技術領域は概ねカバーできている」。目安として10%以下が望ましい。

### クロス集計による注力領域・空白領域の特定
2つの分類軸のクロス集計（フェーズ4のヒートマップ/バブルチャート）から、件数が集中するセル（注力領域）と件数ゼロのセル（空白領域・潜在的な研究機会）を特定できる。

### 出願人別の技術戦略比較
特定の出願人で絞り込んだ分類結果と全体の分類結果を比較することで、各社の技術ポートフォリオの特徴を浮き彫りにできる。例: 「A社はCO2分離膜に偏重(全体比17%に対し同社は35%)しており、膜技術に特化した戦略が見て取れる」。

### 分類ルールの品質評価
ルールの論理式構造（キーワードの網羅性、表記ゆれ対応、近傍演算子の使用有無）から、分類の精度に関する示唆を述べることができる。特にカバレッジが低いカテゴリのルール改善提案が有用である。

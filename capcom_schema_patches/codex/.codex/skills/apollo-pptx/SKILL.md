# APOLLO PPTXスキル — コンサルティングレポート品質のスライド生成（Codex CLI版）

> **Codex CLI 専用版**。Claude Code 用の `.claude/skills/apollo-pptx/SKILL.md` を Codex 仕様に翻案しています。
> 共有資産（`capcom_schema/templates/slides_spec.md`, `apollo_template.pptx`）は ZIP 同梱の `capcom_schema/` 配下をそのまま参照します。

## ⚠️ 優先順位ルール（最重要・厳守）

**スライド生成時は `capcom_schema/templates/slides_spec.md` を最優先で参照すること。**

- 本SKILL.mdの記述（関数名・スライド比率・ポンチ絵パターン数・カラー定義など）と `slides_spec.md` の記述が**矛盾する場合、必ず `slides_spec.md` を採用する**
- 本SKILL.mdは概要・起動条件・運用ルールを示すメタ文書であり、実装仕様の正は `slides_spec.md`（現行 v6.0、スライドタイプ15種）にある
- 関数名は `slides_spec.md` に定義されたもの（`add_chart_slide()` / `add_matrix_slide()` / `add_process_slide()` 等）を使用すること

## 概要
APOLLOのCAPCOMセッションデータ（JSON + スナップショット画像）から、python-pptxを使ってコンサルティングファーム品質のPowerPointスライドを自動生成する。Typst/PDF 本編レポート（`apollo-capcom` スキルの成果物）とは独立した**プレゼン用別冊**として生成できる。

## 起動条件
- ユーザーが「スライドを作って」「PPTを生成して」「プレゼン資料を作って」と依頼
- Codex: `$apollo-pptx` または `/skills` → `apollo-pptx`

## 前提条件
- CAPCOMセッションフォルダ（`session_YYYYMMDD_HHMMSS/`、ZIPを展開したフォルダ）を cwd として作業すること
- `capcom_schema/templates/slides_spec.md` を仕様書として参照すること
- `capcom_schema/templates/apollo_template.pptx` をテンプレートとして使用すること
- `pip install python-pptx Pillow` が必要

## Codex 固有の運用ルール（ツァーリ・ボンバ対策）
- **サブエージェント禁止**: Codex は組込サブエージェントを持たない。全処理をメインコンテキスト内で完結させる
- **ファイル読み込み最小化**: `slides_spec.md` は1回だけ読み、以降は会話内で参照する。`snapshots/` の画像は必要なものだけ読み込む
- **ユーザー確認**: スライド構成（枚数・章立て）の確認が必要な場面では `ask_user_question` ツールでユーザー応答を取得する。`codex exec` 非対話モードでは確認ゲートが機能しないため非推奨
- **patents.csv 全量表示禁止**: `print(df)` / `cat data/patents.csv` は禁止。`head` で構成確認後、pandas でフィルタする

## 設計原則（slides_spec.mdに準拠）

### 可視化ファースト原則
- **チャート・図が主役**、テキストは注釈
- チャート+注釈スライドが全体の50%以上
- テキスト主体（ナラティブ）は10%以下に制限

### タイトル＝結論 原則
- スライドタイトルは**結論そのもの**（ラベルではない）
- `～` で副題を付けて1行でストーリーを完結させる
- 例: 「上位5クラスタが全体の58%を占有 ～技術集中化が加速、差別化領域の特定が急務」

### ■サブメッセージ
- タイトル直下に■マーカー付き1-2行テキスト
- KEY_MSG_BG背景ボックス + 左ACCENTバーで囲む

### フォント
- 欧文/見出し=**Century Gothic**、和文=**Yu Gothic**（フォールバック順は `slides_spec.md` 参照）
- 全runに `lang="ja-JP"` 明示
- 禁則処理適用

### カラーパレット（モノトーン＋クリムゾン単一アクセント）
> 正値は `slides_spec.md` Section 1。青/緑/橙は使わず、白・墨・グレー＋赤で構成する。
```
INK:         #111111  (タイトル、本文強調)
PAPER:       #FFFFFF  (明スライド背景)
DARK_SECTION:#101011  (表紙/章扉/クロージングの黒背景)
ACCENT/RED:  #C51212  (アクセント＝クリムゾン単一)
BRIGHT_RED:  #E33333  (ハイライト)  DEEP_RED: #831010 (チャート最濃)
MEDIUM_GRAY: #686868  (補足・ページ番号)
LIGHT_GRAY:  #F1F2F3 / PALE_GRAY:#F6F7F8 (カード/面)
BORDER_GRAY: #D8DADD  (罫線)
```

### デザインのセンス・ルール（モック検証済み・厳守）
> 詳細と定数は `slides_spec.md` Section 1 原則1/6/7。
- **左マージン固定 `MARGIN_L=0.9`**: タイトル・サブメッセージ・コンテンツを同じ左端に整列。詰め込まず**余白で呼吸させる**（埋めるのは縦方向のみ）。
- **赤は9:1**: クリムゾンは画面の1割未満。**1スライドにつき赤は1〜2点**（結論の数値・1本の罫・注目1要素）。テーブル/チャートも「最も言いたい1点」だけ赤、他はグレー。
- **タイポは3階層**: 見出し / 小見出し・ラベル / 本文。中間サイズを足さない。差はサイズより太さ・字間・色でつける。
- **EYEBROWラベル**: タイトル上に小さな赤ラベル（モジュール名・分類、例 `ATLAS / 出願トレンド`）。`add_title_shape(slide, title, label="...")`。

## スライド構成

### 必須スライド
1. **表紙** — 黒背景、白文字タイトル + Mission Objective + 日付（クリムゾン罫）
2. **エグゼクティブサマリー** — KPI 3-4個 + 結論3点
3. **目次** — 章立て一覧

### 必須締めスライド
- **仮説検証サマリー** — CAPCOMレポートの仮説検証テーブルをスライド化。各仮説の判定（✅/❌/⚠️）と根拠を1行ずつ
- **戦略提言** — プロセスフローで短期→中期→長期の矢印フロー ← **テキストのみ禁止、ポンチ絵必須**
- **Appendix** — データセット概要、分析パラメータ

### テキストのみスライドへのポンチ絵割り当て（⚠️ 必ず適用）

以下のスライドタイプは画像がないためテキストのみになりがち。**必ず対応するポンチ絵を生成すること**（関数名は `slides_spec.md` に従う）:

| スライド | ポンチ絵 |
|--------|--------|
| エグゼクティブサマリー | KPIカード行（9指標を3行×3列） |
| 超領域解説（Saturn V） | 2×2マトリクス（超領域の成長/成熟ポジション） |
| ホワイトスペース | ピラミッド（特許化の機会を層別表示） |
| 主要発見サマリー | プロセスフロー（発見1→2→3→4→5） |
| 推奨アクション | プロセスフロー（短期→中期→長期） |
| 仮説検証 | テーブル（仮説ID/内容/判定/根拠） |
| NEBULAマクロ環境 | タイムライン（政策イベント時系列） |
| 競争構造 | 2×2マトリクス（業種×ポジション） |

## レイアウト連動ルール（重要）

タイトルとサブメッセージの位置は**必ず連動させる**こと。固定座標でハードコードしてはならない。`slides_spec.md` の `add_title_shape()` はタイトルの長さに応じてフォントサイズと高さを動的調整し、サブメッセージの開始y座標を返す。`add_sub_message()` はボックス下端のy座標を返す。コンテンツ（チャート・テーブル等）はその戻り値を起点に配置する。

## 実装手順

1. `capcom_schema/templates/slides_spec.md` を読み込む（全コードパターンが記載）
2. セッションフォルダの `data/` と `snapshots/` と `voyager/mission.json` を確認
3. 利用可能なデータに応じてスライド構成を決定 → `ask_user_question` で枚数・章立てをユーザーに確認
4. `capcom_schema/templates/apollo_template.pptx` を `reports/` にコピーし、slides_spec.md のコードパターンに従って各スライドを生成
5. **各スライドのタイトル→サブメッセージ→コンテンツは必ず戻り値を連鎖させる**
6. 出力先: `reports/` フォルダに `apollo_report_YYYYMMDD.pptx`

## デザインルール（厳守）

### ページ番号（フッター帯は廃止）
- フッター帯・ボトムバー・"APOLLO" ワードマーク・区切り線は**置かない**（紙面はミニマル）
- ページ番号は**右下のみ**。`add_bottom_bar_and_footer(slide, page_num)` が右下にページ番号だけを描画する

### テキスト量の制限
- **サブメッセージ（■ボックス）**: 2行以内（80文字以内）。それ以上は箇条書きに分割する
- **注釈テキスト（チャート横）**: 各■項目は1-2行。箇条書き3-5項目が上限
- **長い文章をそのまま1つのテキストボックスに入れることは禁止**。必ず■マーカーで分割する
- 注釈は14pt、サブメッセージは15ptを厳守

### 図表・ポンチ絵の必須掲載
- **全分析スライド（セクション区切りを除く）に最低1つの視覚要素を含めること**
- テキストのみのスライドは**絶対禁止**（エグゼクティブサマリーでもKPIカードを入れる）
- `snapshots/` に画像がある場合は優先掲載する
- **画像がない章にはポンチ絵を積極的に生成する**（slides_spec.mdのポンチ絵パターンを参照）

### スライド構成比率の厳守
| スライドタイプ | 比率 |
|-------------|------|
| チャート+注釈 | **50%以上** |
| デュアルパネル | 10-15% |
| テーブル+注釈 | 10-15% |
| ナラティブ | **10%以下** |
| セクション区切り | 残り |

### テキストの改行制御
- `word_wrap = True` を全テキストボックスに設定
- `auto_size = MSO_AUTO_SIZE.NONE` でテキストボックスのサイズを固定
- 日本語の禁則処理を全段落に適用
